# 개발 워크플로우 전문가

## 빠른 참조

통합 개발 워크플로우는 TDD, AI 기반 디버깅, 성능 최적화, 자동화 코드 리뷰, 품질 보증을 통합된 워크플로우로 제공

핵심 역량:
- Test-Driven Development: RED-GREEN-REFACTOR 사이클 및 최신 패턴
- AI 기반 디버깅: 지능형 에러 분석 및 솔루션 추천
- 성능 최적화: 프로파일링 및 병목 감지 가이드
- 자동화 코드 리뷰: TRUST 5 검증 프레임워크
- 품질 보증: 포괄적 테스트 및 CI/CD 통합 패턴
- 워크플로우 오케스트레이션: 전체 개발 프로세스 관리

워크플로우 진행: Debug 단계에서 Refactor 단계로, Optimize 단계, Review 단계, Test 단계, Profile 단계 순서로 진행하며 각 단계에서 AI 분석과 추천 제공

사용 시기:
- 완전한 개발 라이프사이클 관리
- 엔터프라이즈급 품질 보증 구현
- 다중 언어 개발 프로젝트
- 성능 중심 애플리케이션
- 기술 부채 감소 이니셔티브
- 자동화 테스트 및 CI/CD 통합

## 구현 가이드

### 핵심 개념

통합 개발 철학:
- 모든 개발 측면을 일관된 워크플로우로 통합
- 복잡한 의사결정을 위한 AI 지원
- 최적의 패턴을 위한 업계 모범 사례 통합
- 워크플로우 단계 간 지속적 피드백 루프
- 자동화된 품질 게이트 및 검증

워크플로우 컴포넌트:

**컴포넌트 1 - AI 기반 디버깅:**
디버깅 컴포넌트는 지능형 에러 분류 및 솔루션 추천 제공. 에러 발생 시 시스템이 에러 유형, 스택 트레이스, 주변 컨텍스트를 분석하여 근본 원인 식별 및 적절한 수정 제안. 현재 모범 사례 및 일반적 에러 해결 패턴 참조

**컴포넌트 2 - 스마트 리팩토링:**
리팩토링 컴포넌트는 기술 부채 분석 및 안전한 자동화 변환 기회 식별. 코드 복잡도, 중복, 유지보수성 메트릭 평가 후 리스크 평가와 함께 구체적 리팩토링 작업 추천

**컴포넌트 3 - 성능 최적화:**
성능 컴포넌트는 실시간 모니터링 가이드 및 병목 감지 제공. CPU 집약 연산, 메모리 누수, I/O 병목 식별 후 식별된 문제에 기반한 구체적 최적화 전략 추천

**컴포넌트 4 - TDD 사이클 관리:**
TDD 컴포넌트는 향상된 테스트 생성으로 RED-GREEN-REFACTOR 사이클 안내. 먼저 실패하는 테스트 작성, 통과를 위한 최소 코드 구현, 테스트 커버리지 유지하며 리팩토링

**컴포넌트 5 - 자동화 코드 리뷰:**
코드 리뷰 컴포넌트는 AI 기반 품질 분석과 함께 TRUST 5 프레임워크 검증 적용. 5개 신뢰 차원에 대해 코드 평가 및 실행 가능한 개선 추천 제공

### TRUST 5 프레임워크

TRUST 5 프레임워크는 5개 차원을 가진 개념적 품질 평가 모델. 이 프레임워크는 구현된 모듈이 아닌 코드 품질 평가 지침 제공

**차원 1 - 테스트 가능성 (Testability):**
코드가 효과적으로 테스트될 수 있는지 평가. 고려사항: 함수가 순수하고 결정적인가? 의존성 주입 가능한가? 단위 테스트에 충분히 모듈화되었는가? 점수는 상당한 리팩토링이 필요한 낮은 테스트 가능성부터 우수한 테스트 커버리지 지원의 높은 테스트 가능성까지 범위

**차원 2 - 가독성 (Readability):**
다른 개발자가 코드를 얼마나 쉽게 이해할 수 있는지 평가. 고려사항: 변수 및 함수 이름이 설명적인가? 코드 구조가 논리적인가? 복잡한 연산이 문서화되었는가? 점수는 명명 규칙, 코드 구성, 문서화 품질 평가

**차원 3 - 이해 가능성 (Understandability):**
구현의 개념적 명확성 평가. 고려사항: 비즈니스 로직이 명확하게 표현되었는가? 추상화가 적절한가? 새 개발자가 코드를 빠르게 이해할 수 있는가? 표면 가독성을 넘어 아키텍처 명확성 평가

**차원 4 - 보안 (Security):**
보안 상태 및 취약점 노출 평가. 고려사항: 입력이 검증되는가? 비밀이 적절히 관리되는가? 일반적 취약점 패턴(인젝션, XSS, CSRF)이 회피되는가? 점수는 보안 모범 사례 준수 평가

**차원 5 - 투명성 (Transparency):**
운영 가시성 및 디버그 가능성 평가. 고려사항: 에러 처리가 포괄적인가? 로그가 의미 있고 구조화되었는가? 시스템을 통해 문제 추적 가능한가? 점수는 관찰 가능성 및 문제 해결 역량 평가

**전체 TRUST 점수 계산:**
전체 TRUST 점수는 가중 평균을 사용하여 5개 차원 결합. 어떤 차원에서든 심각한 문제가 평균을 오버라이드하여 보안 또는 테스트 가능성 문제가 다른 곳의 높은 점수로 가려지지 않도록 보장. 통과 점수는 일반적으로 각 차원의 최소 임계값과 허용 가능한 가중 평균 필요

### 기본 워크플로우 구현

**디버깅 워크플로우 프로세스:**
- 단계 1: 스택 트레이스, 환경, 최근 코드 변경을 포함한 전체 컨텍스트로 에러 캡처
- 단계 2: 에러 유형 분류 (구문, 런타임, 로직, 통합, 성능)
- 단계 3: 알려진 이슈 데이터베이스 및 모범 사례에 대해 에러 패턴 분석
- 단계 4: 성공 가능성에 따라 순위가 매겨진 솔루션 후보 생성
- 단계 5: 추천 수정 적용 및 해결 확인
- 단계 6: 향후 참조를 위해 이슈 및 솔루션 문서화

**리팩토링 워크플로우 프로세스:**
- 단계 1: 코드 스멜 및 기술 부채 지표에 대해 대상 코드베이스 분석
- 단계 2: 순환 복잡도 및 결합도를 포함한 복잡도 메트릭 계산
- 단계 3: 관련 리스크 수준과 함께 리팩토링 기회 식별
- 단계 4: 우선순위가 지정된 작업으로 리팩토링 계획 생성
- 단계 5: 안전한 증분으로 리팩토링 변환 적용
- 단계 6: 테스트 실행을 통한 동작 보존 확인

**성능 최적화 프로세스:**
- 단계 1: 대상 메트릭(CPU, 메모리, I/O, 네트워크)에 대한 프로파일링 구성
- 단계 2: 대표적 부하 조건에서 프로파일링 실행
- 단계 3: 병목 식별을 위한 프로파일링 결과 분석
- 단계 4: 예상 영향 추정과 함께 최적화 추천 생성
- 단계 5: 개별 효과 측정을 위해 격리된 최적화 적용
- 단계 6: 전체 성능 향상 검증

**TDD 사이클 프로세스:**
- RED 단계: 원하는 동작을 정의하는 실패하는 테스트 작성. 테스트는 코드가 무엇을 해야 하는지 명확히 표현해야 하며, 어떻게 해야 하는지는 아님. 예상된 이유로 실패하는지 확인하기 위해 테스트 실행
- GREEN 단계: 테스트를 통과하기 위해 필요한 최소 코드 작성. 과도한 엔지니어링 또는 조기 최적화 회피. 테스트 요구사항 만족에만 집중
- REFACTOR 단계: 모든 테스트가 통과하는 상태에서 코드 구조 개선. 디자인 패턴 적용, 공통 기능 추출, 네이밍 개선. 각 리팩토링 단계 후 테스트 실행

**코드 리뷰 프로세스:**
- 단계 1: 리뷰가 필요한 파일 식별을 위한 코드베이스 스캔
- 단계 2: 각 파일에 TRUST 5 프레임워크 분석 적용
- 단계 3: 즉각적 주의가 필요한 중요 이슈 식별
- 단계 4: 파일별 및 집계 품질 점수 계산
- 단계 5: 우선순위 순위와 함께 실행 가능한 추천 생성
- 단계 6: 개선 로드맵과 함께 요약 보고서 생성

### 일반적 사용 사례

**엔터프라이즈 개발 워크플로우:**
엔터프라이즈 애플리케이션의 경우 워크플로우는 각 단계에 품질 게이트 통합. 배포 전 코드는 최소 TRUST 점수 임계값 통과, 0개의 중요 이슈 식별, 필요한 테스트 커버리지 비율 충족 필요. 품질 게이트 구성은 최소 신뢰 점수(일반적으로 0.85), 최대 허용 중요 이슈(일반적으로 0개), 필요한 커버리지 수준(일반적으로 80%) 지정

**성능 중심 애플리케이션:**
성능에 민감한 시스템의 경우 워크플로우는 프로파일링 및 최적화 단계 강조. 성능 임계값은 최대 허용 응답 시간, 메모리 사용량 제한, 최소 처리량 요구사항 정의. 워크플로우는 백분율 개선 추적 및 구체적 최적화 추천 제공

## 고급 기능

### 워크플로우 통합 패턴

**지속적 통합 연동:**
워크플로우는 다단계 검증 프로세스를 통해 CI/CD 파이프라인과 통합. CI 파이프라인은 다음 단계를 순차적으로 실행:

- 단계 1 - 코드 품질 검증: 코드 리뷰 컴포넌트 실행 및 결과가 품질 표준 충족 확인. 품질 검사 실패 시 파이프라인은 품질 실패 보고서와 함께 종료
- 단계 2 - 테스트 검증: 단위, 통합, e2e 테스트를 포함한 전체 테스트 스위트 실행. 테스트 실패 시 파이프라인은 테스트 실패 보고서와 함께 종료
- 단계 3 - 성능 검증: 성능 테스트 실행 및 정의된 임계값과 결과 비교. 성능 표준 미달 시 파이프라인은 성능 실패 보고서와 함께 종료
- 단계 4 - 보안 검증: 정적 분석 및 의존성 스캔을 포함한 보안 분석 실행. 중요 취약점 발견 시 파이프라인은 보안 실패 보고서와 함께 종료

모든 단계 통과 시 파이프라인은 성공 보고서 생성 및 배포 진행

### 품질 게이트 구성

품질 게이트는 각 워크플로우 단계에서 충족해야 할 기준 정의. 게이트는 다양한 엄격도 수준으로 구성 가능:

**엄격 모드:** 모든 품질 차원이 임계값 이상 충족 필요. 모든 중요 이슈가 진행 차단. 전체 테스트 커버리지 요구사항 적용

**표준 모드:** 평균 품질 점수가 임계값 충족 필요. 중요 이슈는 진행 차단하지만 경고 허용. 표준 커버리지 요구사항 적용

**관용 모드:** 중요 차단 이슈만 진행 방지. 품질 점수는 경고 생성하지만 차단하지 않음. 감소된 커버리지 요구사항 적용

게이트 구성은 각 TRUST 차원의 임계값, 심각도별 최대 허용 이슈, 필요 테스트 커버리지 수준, 성능 벤치마크 목표 포함

### 다중 언어 지원

워크플로우는 여러 프로그래밍 언어에서 개발 지원. 언어별 적응 포함:

**Python 프로젝트:** pytest 테스트, pylint 및 flake8 정적 분석, bandit 보안 스캔, cProfile 또는 memory_profiler 성능 분석과 통합

**JavaScript/TypeScript 프로젝트:** Jest 또는 Vitest 테스트, ESLint 정적 분석, npm audit 보안 스캔, Chrome DevTools 또는 lighthouse 성능 분석과 통합

**Go 프로젝트:** go test 테스트, golint 및 staticcheck 정적 분석, gosec 보안 스캔, pprof 성능 분석과 통합

**Rust 프로젝트:** cargo test 테스트, clippy 정적 분석, cargo audit 보안 스캔, flamegraph 성능 분석과 통합

## 기술 스택 참조

워크플로우는 각 역량 영역에 업계 표준 도구 활용:

분석 라이브러리:
- cProfile: Python 프로파일링 및 성능 분석 제공
- memory_profiler: 메모리 사용량 분석 및 최적화 활성화
- psutil: 시스템 리소스 모니터링 지원
- line_profiler: 라인별 성능 프로파일링 제공

정적 분석 도구:
- pylint: 포괄적 코드 분석 및 품질 검사 수행
- flake8: 스타일 가이드 준수 및 에러 감지 강제
- bandit: 보안 취약점 스캔
- mypy: 정적 타입 검증

테스트 프레임워크:
- pytest: 픽스처 및 플러그인과 함께 고급 테스트 제공
- unittest: 표준 라이브러리 테스트 역량 제공
- coverage: 코드 커버리지 측정 및 테스트되지 않은 경로 식별

## 통합 패턴

### GitHub Actions 통합

워크플로우는 다단계 작업 구성을 통해 GitHub Actions와 통합:

작업 구성 단계:
- 단계 1: actions/checkout 사용하여 저장소 체크아웃
- 단계 2: actions/setup-python 및 대상 Python 버전으로 Python 환경 설정
- 단계 3: 테스트 및 분석 도구 포함 프로젝트 의존성 설치
- 단계 4: 엄격한 품질 게이트로 품질 검증 워크플로우 실행
- 단계 5: 커버리지 보고와 함께 테스트 스위트 실행
- 단계 6: 베이스라인 메트릭 대비 성능 벤치마킹 수행
- 단계 7: 보안 스캔 및 취약점 감지 실행
- 단계 8: 리뷰를 위해 워크플로우 결과를 작업 아티팩트로 업로드

작업은 push 및 pull request 이벤트에서 실행하도록 구성 가능하며, 필요시 여러 Python 버전에서 매트릭스 테스트 가능

### Docker 통합

컨테이너화된 환경의 경우 워크플로우는 Docker 컨테이너 내에서 실행:

컨테이너 구성:
- 최소 크기를 위해 Python slim 변형 기반 이미지
- requirements 파일에서 프로젝트 의존성 설치
- 프로젝트 소스 코드를 컨테이너에 복사
- 완전한 워크플로우 시퀀스 실행을 위한 entrypoint 구성
- 영구 저장소가 필요한 경우 결과 출력을 위한 볼륨 마운트

컨테이너화된 워크플로우는 개발, 테스트, 프로덕션 시스템 전반에 걸쳐 일관된 실행 환경 보장

do-workflow execute --project /project/src --mode full

개별 컴포넌트 실행:
do-workflow debug --file app.py --error "AttributeError"
do-workflow refactor --directory src/ --max-risk medium
do-workflow profile --target function_name --types cpu,memory
do-workflow test --spec user_auth.spec --mode tdd
do-workflow review --project /project/src --trust-score-min 0.8

CI 모드:
do-workflow ci --commit abc123 --quality-gates strict
