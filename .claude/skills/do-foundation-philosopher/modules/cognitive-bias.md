# Cognitive Bias Module

기술 의사결정에서 인지 편향 식별 및 완화에 대한 심층 가이드.

## 기술 의사결정의 주요 편향

### Anchoring Bias

정의: 처음 접한 정보에 과도하게 의존.

개발에서 나타나는 방식:
- 처음 고려한 솔루션이 기본값이 됨
- 초기 추정치가 기준선이 됨
- 처음 조사한 기술이 선호를 받음
- 변경되어도 원래 요구사항이 지배적

탐지 질문:
- 이 솔루션이 선호되는 이유가 먼저여서인가 최선이어서인가?
- 대안을 먼저 접했다면 같은 결론에 도달할까?
- 초기 값에서 조정하고 있나, 독립적으로 평가하고 있나?

완화 전략:
- 커밋 전에 의도적으로 대안 고려
- 비교 전에 독립적으로 옵션 생성
- 다른 정보로 시작한 사람들에게 의견 구함
- 구조화된 평가 프레임워크 사용

### Confirmation Bias

정의: 기존 믿음을 확인하는 정보를 찾고 해석.

개발에서 나타나는 방식:
- 선호하는 기술의 성공 사례 검색
- 선택한 접근법에 대한 우려 무시
- 모호한 데이터를 의사결정 지지로 해석
- 긍정적 피드백에 선택적 주의

탐지 질문:
- 내 선호에 도전하는 증거를 찾고 있는가?
- 내 견해에 반대하는 증거를 수용할 것인가?
- 누군가 왜 동의하지 않을 수 있는지 진정으로 고려했는가?

완화 전략:
- 적극적으로 반례 검색
- 누군가에게 반대 입장을 논증하도록 할당
- 선호하는 옵션이 실패할 수 있는 이유 나열
- 알려진 회의론자에게 피드백 요청

### Sunk Cost Fallacy

정의: 이전에 투자한 리소스 때문에 계속 투자.

개발에서 나타나는 방식:
- 투자한 시간 때문에 실패하는 접근법 계속
- 과거 노력 때문에 레거시 코드 유지
- 교육 투자 때문에 도구 고수
- 공개적 약속 때문에 의사결정 방어

탐지 질문:
- 오늘 처음 시작한다면 이 경로로 갈 것인가?
- 미래 가치 때문에 계속하는가 과거 투자 때문인가?
- 이 상황의 다른 사람에게 무엇을 조언할 것인가?

완화 전략:
- 미래 비용과 이익만 기반으로 의사결정 평가
- 시작 전에 중단 기준을 명확히 설정
- 신선한 관점으로 정기 체크포인트 리뷰
- 과거 투자와 의사결정을 명시적으로 분리

### Availability Heuristic

정의: 쉽게 떠오르는 정보에 과도한 비중.

개발에서 나타나는 방식:
- 최근 프로젝트 경험이 사고를 지배
- 기억에 남는 실패가 과도한 주의를 유발
- 익숙한 기술이 더 나은 대안보다 선호됨
- 최근 뉴스가 기술 평가에 영향

탐지 질문:
- 이 우려가 빈도 기반인가 기억하기 쉬움 기반인가?
- 체계적 데이터 vs 일화적 경험이 말하는 바는?
- 최근 사건에 과도한 비중을 두고 있지 않은가?

완화 전략:
- 객관적 비교를 위해 base rate 데이터 탐색
- 최근 기억 너머의 경험 고려
- 회상보다 문서와 연구 참조
- 개인 경험과 더 넓은 증거 균형

### Overconfidence Bias

정의: 자신의 판단과 예측에 과도한 확신.

개발에서 나타나는 방식:
- 구현 복잡성 과소평가
- 엣지 케이스 처리 능력 과대평가
- 사용자 행동 예측에 확신
- 불확실성 범위 없는 추정치 확신

탐지 질문:
- 유사 예측에 대한 내 실적은?
- 이것이 다르게 밝혀지면 어떤 기분일까?
- 내가 틀리려면 무엇이 참이어야 하나?

완화 전략:
- 추정치를 신뢰 수준이 있는 범위로 표현
- 예측의 역사적 정확도 검토
- 중요한 판단에 외부 검증 요청
- 권고에 명시적 불확실성 포함

## 편향 완화 기법

### Pre-mortem 분석

프로세스:
1. 의사결정이 구현되었다고 상상
2. 참담하게 실패했다고 가정
3. 실패의 모든 이유를 기록
4. 가장 가능성 높은 실패 모드 평가
5. 최고 위험 실패 모드를 사전에 해결

이점:
- 이의 제기와 우려 표명을 정당화
- 낙관주의가 숨길 수 있는 위험 표면화
- 구체적 완화 기회 생성

### Devil's Advocate

프로세스:
1. 누군가에게 선호 옵션에 반대 논증 할당
2. 가능한 가장 강력한 사례를 만들도록 요구
3. 반대 논증을 진지하게 받아들임
4. 반대 논증이 blind spot을 드러내면 의사결정 조정

가이드라인:
- Devil's advocate가 철저히 준비해야 함
- 논증이 형식적이 아닌 실질적이어야 함
- 응답이 논증을 해결해야지 무시하면 안 됨

### Outside View

프로세스:
1. 유사 의사결정의 참조 클래스 식별
2. 성공과 실패의 base rate 조사
3. 현재 상황을 참조 클래스와 비교
4. base rate 기반으로 확신 조정

질문:
- 역사적으로 유사 의사결정은 어떻게 되었나?
- 이 유형의 프로젝트의 일반적 성공률은?
- 이 상황이 참조 클래스와 다른 점은?

## Bias Check 체크리스트

중요한 권고를 최종화하기 전에:

Anchoring Check:
- 첫 아이디어에 정착하기 전에 대안을 고려했는가
- anchor에 상대적이 아닌 독립적으로 옵션을 평가했는가
- 다른 시작점을 가진 사람들의 관점을 구했는가

Confirmation Check:
- 선호 옵션에 반하는 증거를 적극적으로 찾았는가
- 반대 논증을 진정으로 고려했는가
- 회의론자나 비평가와 상담했는가

Sunk Cost Check:
- 미래 가치만 기반으로 평가했는가
- 새로 온 사람에게 무엇을 조언할지 고려했는가
- 과거 투자와 의사결정을 분리했는가

Availability Check:
- 개인 경험 너머 base rate를 확인했는가
- 최근 사건 너머 데이터를 고려했는가
- 일화와 체계적 증거를 균형 있게 했는가

Overconfidence Check:
- 예측에 불확실성을 표현했는가
- 내가 틀린 시나리오를 고려했는가
- 유사 예측의 역사적 정확도를 검토했는가

## 의사결정 프로세스 통합

편향 점검 타이밍:
- 초기 권고 생성 후
- 이해관계자에게 옵션 제시 전
- 입장을 강하게 방어할 때
- 대안을 빠르게 기각할 때

문서화:
- 어떤 편향을 점검했는지 기록
- 식별된 편향과 해결 방법 기록
- 남은 불확실성 문서화
